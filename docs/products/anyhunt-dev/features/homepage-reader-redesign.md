## 1. 需求背景

### 1.1 当前问题

| 问题             | 描述                                                               |
| ---------------- | ------------------------------------------------------------------ |
| **官网思维**     | 首页是传统的产品介绍页，Hero 区域占满整屏，用户看不到内容          |
| **路径过长**     | 用户需要「首页 → 注册 → 登录 → Dashboard → Subscriptions」才能使用 |
| **功能分散**     | 订阅管理、Inbox、Topic 发布分散在不同页面，体验割裂                |
| **登录前后割裂** | 未登录看营销页，登录后看 Dashboard，界面完全不同                   |

### 1.2 目标状态

将首页从「官网模式」转变为「智能订阅平台」：

- **打开即内容**：用户访问 anyhunt.app 直接进入内容消费界面
- **功能整合**：订阅管理、Inbox、配置都在同一界面完成
- **统一体验**：登录前后界面结构一致
- **Notion 风格交互**：hover 显示操作、右键菜单、弹窗配置

### 1.3 Slogan

**"Get anything with AI"**

## 2. 产品定位

### 2.1 核心价值

Anyhunt Digest **不是** RSS 阅读器，而是 **AI 驱动的智能订阅平台**：

| 传统 RSS 阅读器  | Anyhunt Digest                               |
| ---------------- | -------------------------------------------- |
| 订阅现有 Feed    | **用户自定义关键词**，AI 自动抓取            |
| 被动接收所有内容 | AI 评分筛选，只推送高质量内容                |
| 无个性化         | **反馈学习**：保存/不感兴趣 → 系统越用越懂你 |
| 只能自己看       | 可**发布为 Topic** 供他人订阅                |

### 2.2 核心功能（必须保留）

| 功能           | 说明                                  | 入口位置              |
| -------------- | ------------------------------------- | --------------------- |
| **创建订阅**   | 输入关键词 + 配置参数，AI 定期抓取    | 左侧面板 [+] 按钮     |
| **订阅配置**   | 评分阈值、抓取数量、投递渠道、AI 摘要 | 订阅项右键菜单 → 弹窗 |
| **手动运行**   | 立即触发一次抓取                      | 配置弹窗 / 右键菜单   |
| **运行历史**   | 查看每次抓取的结果                    | 配置弹窗 Tab          |
| **反馈学习**   | 系统学习用户偏好，提供优化建议        | 配置弹窗 Tab          |
| **发布 Topic** | 将订阅发布为公开 Topic                | 配置弹窗 / 右键菜单   |
| **Inbox 操作** | 标记已读、保存、不感兴趣              | 文章详情操作栏        |
| **统计概览**   | 未读数、订阅数、已保存数              | 左侧面板顶部          |

### 2.3 登录状态对比

| 维度     | 未登录                     | 已登录                    |
| -------- | -------------------------- | ------------------------- |
| 左侧面板 | 推荐 Topic（只读）         | 用户的订阅列表 + 操作能力 |
| 内容区域 | 推荐 Topic 的内容          | 用户 Inbox 内容           |
| 操作能力 | 只能浏览，点击操作跳转登录 | 完整功能                  |
| 界面结构 | **完全一致**               | **完全一致**              |

## 3. 信息架构

### 3.1 整体架构

```
anyhunt.app（唯一入口）
│
├── 左侧面板（200px）
│   │
│   ├── 头像区域
│   │   ├── 头像 + 用户名（点击展开菜单）
│   │   └── 统计徽章：3 未读 · 5 订阅
│   │
│   ├── ════════════════════════════
│   │
│   ├── 🔍 发现                      ← 固定入口
│   │
│   ├── ════════════════════════════
│   │
│   ├── 📥 全部                      ← 所有订阅的混合 Inbox
│   ├── ⭐ 已保存                    ← 筛选已保存
│   │
│   ├── ────────────────────────────
│   │
│   ├── [+] 新建订阅                 ← 创建按钮
│   │
│   ├── ────────────────────────────
│   │
│   └── 我的订阅                     ← 用户订阅列表
│       ├── AI 行业动态          ⋯   ← hover 显示更多菜单
│       ├── 竞品监控             ⋯
│       ├── 技术周刊             ⋯
│       └── ...
│
├── 中间栏（320px）：文章列表
│   ├── 顶部：订阅名 + ⟳ 刷新 + ▾ 筛选 + ☑ 全部已读
│   └── 文章卡片列表
│
└── 右侧栏（flex）：文章详情
    ├── 顶部：标题 + ⭐ 保存 + 👎 不感兴趣 + [↗] 打开原文
    ├── 来源 + 时间
    ├── AI 摘要卡片
    └── 正文内容（Markdown 渲染）
```

### 3.2 页面路由

| 路由         | 说明        | 内容                           |
| ------------ | ----------- | ------------------------------ |
| `/`          | 首页/阅读器 | 三栏布局，默认显示全部 Inbox   |
| `/discover`  | 发现页面    | 浏览公开 Topic、创建自定义订阅 |
| `/developer` | 开发者页面  | API 文档入口                   |

> 注：所有功能通过弹窗完成，不跳转独立页面

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│             │    │             │    │             │
│   第一栏    │ ←→ │   第二栏    │ ←→ │   第三栏    │

│  订阅列表   │    │  文章列表   │    │  文章详情   │

└─────────────┘    └─────────────┘    └─────────────┘

      ←── 左滑返回上一级 ──        ←── 左滑返回上一级 ──

```

### 7.4 文章详情渲染逻辑

**核心原则**：右侧详情区域展示**本地渲染的 Markdown 内容**，而非嵌入原网页。

```
用户点击文章
       ↓
调用 Fetchx API 抓取（返回 Markdown）
       ↓
前端渲染 Markdown + 显示操作栏
```

**技术实现**：

| 步骤     | 说明                                                              |
| -------- | ----------------------------------------------------------------- |
| 内容抓取 | 调用 Fetchx API（`/api/v1/scrape`），指定 `formats: ['markdown']` |
| 前端渲染 | 使用 react-markdown 渲染 Markdown 内容                            |
| 原文跳转 | 详情区右上角 [↗] icon，点击新开 Tab                               |

> 注：不在客户端直接用 Readability 是因为浏览器端 fetch 会遇到 CORS 限制。

#### 全文内容获取问题详述

**已实现部分**：

- 后端 `GET /api/v1/app/digest/inbox/:id/content` 端点
- 前端 `useInboxItemContent` hook
- `ArticleDetail` 组件支持显示全文

**待解决问题**：
当前 `ContentService.getFulltext()` 只从 Redis 缓存获取数据（TTL 7 天），缓存 miss 时返回 `null`。

缓存 miss 场景：

1. 文章投递超过 7 天 → 缓存过期
2. 入池时没有 fulltext（某些 RSS 只提供摘要）
3. Redis 重启/数据丢失

**可选方案**：

| 方案                | 策略                                      | 优点           | 缺点                             |
| ------------------- | ----------------------------------------- | -------------- | -------------------------------- |
| A. 实时抓取         | 缓存 miss 时调用 Fetchx 重新抓取          | 总能获取内容   | 延迟高、消耗配额、页面可能已变化 |
| B. 持久化到 DB      | 全文存 PostgreSQL 而非 Redis              | 永久保存       | 存储成本高、全文可能很大         |
| C. 优雅降级         | 缓存 miss 时返回 null，前端只显示 AI 摘要 | 实现简单       | 用户体验不完整                   |
| D. 混合策略（推荐） | 缓存优先 + 按需抓取按钮                   | 平衡体验和成本 | 逻辑稍复杂                       |

**推荐方案 D 流程**：

```
用户打开文章
    ↓
1. 优先显示 AI 摘要（立即可见）
2. 并行请求全文缓存
    ↓
缓存命中 → 显示全文
缓存 miss → 显示提示 + "加载原文"按钮
              ↓ 用户点击
          调用 Fetchx 实时抓取 → 显示并更新缓存
```
